## 一、TapNow（tapnow.ai）深度对标

### 1.1 产品定位与核心架构

| 维度 | TapNow实现 | StarCanvas差距 | 优先级 |
|------|-----------|---------------|--------|
| **产品定位** | AI商业视觉创作引擎，"ComfyUI + Photoshop"的融合体验 | 需明确是否为同一赛道 | P0 |
| **核心交互** | 无限缩放画布 + 节点编辑（类似ComfyUI） | 需评估当前画布架构 | P0 |
| **技术架构** | 浏览器端渲染 + 云端AI模型聚合 | 需确认服务端架构 | P0 |
| **商业模式** | 社区(TapTV) + 工作流市场 + 会员订阅 | 需规划商业模式 | P1 |

### 1.2 Cinema Lab 交互深度解析

| 功能点 | TapNow实现细节 | 技术方案推测 | StarCanvas差距 |
|--------|---------------|-------------|---------------|
| **多角度拖拽** | 支持在画布上拖拽调整镜头角度、景别，实时预览效果 | 前端Canvas/WebGL渲染 + 后端AI重绘API | 需确认是否支持多视角实时预览 |
| **火柴人（Pose）** | 可拖拽火柴人调整角色姿态，AI根据姿态生成对应画面 | 姿态检测模型（OpenPose/ControlNet）+ 骨骼映射算法 | 需集成ControlNet或类似方案 |
| **魔法棒焦点编辑** | 点击画面区域自动识别焦点，支持焦点深度、虚化程度调整 | 深度估计模型（MiDaS/DPT）+ 后期处理管线 | 需集成深度估计能力 |
| **画布缩放** | 无限缩放画布，支持节点自由布局 | 基于变换矩阵的Canvas缩放（CSS transform + 虚拟坐标系） | 评估现有Canvas实现 |

**可直接借鉴的设计模式**:
- **无限画布模式**: 使用`CSS transform: scale()` + 虚拟坐标映射，参考Excalidraw、Figma的实现
- **节点编辑器模式**: 参考ComfyUI的节点连线逻辑（React Flow / Rete.js框架）
- **AI预览管线**: 采用"请求队列 + 进度推送（SSE/WebSocket）"模式

### 1.3 Chat操作画布机制

| 功能点 | TapNow实现细节 | 技术方案推测 | StarCanvas差距 |
|--------|---------------|-------------|---------------|
| **AI创建节点** | 自然语言描述 → AI解析意图 → 自动生成对应节点 | LLM（GPT-4/Claude）+ Function Calling/结构化输出 | 需集成LLM节点生成能力 |
| **连接节点** | AI自动识别节点间依赖关系，自动连线 | 图结构分析（DAG拓扑排序）+ 类型匹配算法 | 需实现自动拓扑连接 |
| **Prompt格式** | 用户输入 → AI返回JSON格式的节点配置 | 结构化JSON Schema定义节点类型和参数 | 需定义StarCanvas节点Schema |
| **返回格式** | 包含节点类型、位置坐标、参数值、连接关系 | JSON格式，包含`nodes[]`和`edges[]`数组 | 需设计兼容的DSL |

**Prompt工程参考（推测）**:
```json
{
  "nodes": [
    {
      "id": "text-1",
      "type": "TextPrompt",
      "position": { "x": 100, "y": 200 },
      "data": { "prompt": "a cinematic portrait of..." }
    },
    {
      "id": "img-1",
      "type": "ImageGen",
      "position": { "x": 400, "y": 200 },
      "data": { "model": "sd-xl", "steps": 30 }
    }
  ],
  "edges": [
    { "source": "text-1", "target": "img-1", "type": "prompt" }
  ]
}
```

### 1.5 角色三视图锁定技术方案

## 二、小云雀短剧Agent 2.0 深度对标

### 2.1 产品定位与核心架构

| 维度 | 小云雀实现 | StarCanvas差距 | 优先级 |
|------|-----------|---------------|--------|
| **产品定位** | 面向短剧创作的AI Agent，从剧本到成片一站式 | 需评估是否覆盖短剧赛道 | P0 |
| **核心架构** | Agent自主规划 + 多Agent协作 + 工具调用 | 需确认Agent架构设计 | P0 |

### 2.2 100+影视级画风库实现

| 功能点 | 小云雀实现细节 | 技术方案 | StarCanvas差距 |
|--------|---------------|---------|---------------|
| **画风分类** | 按场景分类：都市/古装/悬疑/科幻/甜宠等 | 每个画风对应一组LoRA + VAE + 提示词模板 | 需建立画风模型库 |
| **画风切换** | 一键切换画风，自动应用到全部分镜 | 全局风格注入（Style Adapter / IP-Adapter） | 需实现全局风格锁 |
| **预览机制** | 切换前显示画风样例图 | 预生成画风样例缓存 | 需预生成参考图 |
| **自定义画风** | 支持用户上传参考图训练自定义画风 | LoRA训练 + 画风提取算法 | 需集成训练管线 |

**技术实现细节**:
- 每个画风 = 1个LoRA（约50-150MB）+ 1组默认提示词前缀 + 1个VAE配置
- 画风管理使用数据库存储：画风ID、名称、分类、LoRA路径、提示词模板、样例图URL
- 加载时自动注入正向/反向提示词前缀

**开源可复用方案**:
- **画风模型**: CivitAI热门LoRA可直接复用（需版权审查）
- **画风管理**: 参考Stable Diffusion WebUI的Prompt Style系统
- **预训练**: 使用`kohya-ss`脚本批量训练画风LoRA

### 2.3 参数化独立面板（光影/镜头/景别）

| 面板 | 参数项 | 到Prompt的映射逻辑 | StarCanvas差距 |
|------|--------|-------------------|---------------|
| **光影面板** | 主光源方向（8方向）、强度（0-100%）、色温（2700K-6500K）、软硬光 | 方向→"side lighting/back lighting"；色温→"warm light/cool light"；软硬→"soft/hard lighting" | 需建立参数→提示词映射表 |
| **镜头面板** | 焦距（24mm-200mm）、光圈（f/1.4-f/16）、运镜（推/拉/摇/移/跟） | 焦距→"wide angle/telephoto lens"；光圈→"shallow/deep depth of field"；运镜→"dolly in/dolly out" | 需建立镜头术语映射 |
| **景别面板** | 特写/近景/中景/全景/远景、画幅比（16:9/9:16/1:1/2.35:1） | 景别→"close-up/medium shot/long shot"；画幅→比例参数 | 需实现构图自动调整 |

**设计模式建议**:
- **参数DSL**: 定义一套JSON格式的参数描述语言，前端渲染为UI控件
- **A/B对比预览**: 调整参数后并排显示前后效果
- **参数预设**: 支持保存常用参数组合（如"夜晚追光"、"日景柔光"）

### 2.4 「@」资产调用系统技术方案

| 功能点 | 小云雀实现细节 | 技术方案 | StarCanvas差距 |
|--------|---------------|---------|---------------|
| **资产引用** | 在Prompt中输入`@角色名`或`@场景名`自动引用 | 前端输入监听（@触发）→ 弹出资产选择器 → 插入资产ID | 需实现@触发选择器 |
| **资产类型** | 角色、场景、道具、服装、表情等 | 资产分类体系 + 向量数据库检索 | 需建立资产分类体系 |
| **资产注入** | 引用后自动将资产特征注入到生成流程 | 资产特征编码 → 注入到IP-Adapter/LoRA触发词 | 需实现资产特征注入 |
| **资产管理** | 个人资产库 + 团队共享资产库 | RBAC权限 + 资产版本管理 | 需实现资产权限管理 |

**技术实现建议**:
- 前端：使用`contenteditable`或`textarea` + `@`符号监听，类似飞书/钉钉的@人功能
- 后端：资产存储使用对象存储（OSS/S3）+ 元数据存MySQL/PostgreSQL
- 检索：资产名称全文检索 + 标签检索
- 注入：将资产对应的LoRA/Embedding自动追加到提示词

### 2.6 剪映草稿导出格式

| 功能点 | 小云雀实现细节 | 技术方案 | StarCanvas差距 |
|--------|---------------|---------|---------------|
| **导出格式** | 生成剪映可导入的草稿文件（.draft或JSON） | 逆向工程剪映草稿格式 + 按格式生成 | 需研究剪映草稿结构 |
| **导出内容** | 分镜图序列、配音音频、字幕、转场效果 | 将分镜数据映射到剪映时间线格式 | 需定义分镜→时间线映射 |

### 2.7 角色一致性锁定机制

| 功能点 | 小云雀实现细节 | 技术方案 | StarCanvas差距 |
|--------|---------------|---------|---------------|
| **一致性方案** | 角色锁定后，所有分镜保持同一角色外观 | 核心方案：InstantID/IP-Adapter + 人脸增强（GFPGAN/CodeFormer） | 需集成一致性方案 |
| **面部锁定** | 基于参考图保持面部特征一致 | 人脸特征向量提取（ArcFace）+ 相似度约束 | 需集成人脸特征模型 |
| **服装锁定** | 保持服装款式和颜色一致 | 服装区域分割（SAM）+ 颜色直方图匹配 | 需实现服装锁定 |
| **动态调整** | 支持微调角色表情、姿态但保持身份 | 身份特征与表情特征解耦（Disentangled Representation） | 需实现特征解耦 |

**开源可复用代码**:
- **身份保持**: `InstantID`（最佳实践）、`IPAdapter-FaceID`
- **人脸增强**: `GFPGAN`、`CodeFormer`
- **换脸/一致性**: `ROOP`、`ReActor`（ComfyUI节点）

## 三、LibreTV 深度对标与MAM方案纠正

### 3.1 LibreTV 真实定位解析

**重要纠正**: LibreTV不是影视资产管理（MAM）系统，而是**视频聚合播放平台**。

| 维度 | LibreTV实现 | 说明 |
|------|-----------|------|
| **产品定位** | 免费开源的视频聚合播放网站 | 类似早期"电影站"的技术方案 |
| **核心功能** | 多视频源聚合搜索、在线播放、倍速、广告过滤 | 不涉及内容管理、编目、转码等MAM功能 |
| **技术栈** | HTML + CSS3 + JavaScript 单页面应用 | 纯前端实现，无后端 |
| **视频源** | 聚合第三方视频源（搜索解析播放） | 不涉及自有媒体资产管理 |
| **部署方式** | 静态页面托管（Vercel/Cloudflare Pages/自有服务器） | 无需数据库 |

**技术架构**:
```
LibreTV/
├── index.html          # 主页面入口
├── player.html         # 播放器页面
├── css/
│   └── style.css       # 样式表
├── js/
│   ├── app.js          # 主应用逻辑
│   ├── api.js          # 视频源API接口
│   ├── config.js       # 配置文件
│   └── ui.js           # UI交互逻辑
└── readme.md
```

### 3.2 LibreTV核心功能矩阵

| 功能 | 实现细节 | 技术方案 | 与StarCanvas关系 |
|------|---------|---------|----------------|
| **多源搜索** | 聚合多个视频源的搜索结果 | 前端并发请求多个API → 合并去重 → 统一展示 | 可作为素材搜索参考 |
| **在线播放** | 调用DPlayer/ArtPlayer播放m3u8/mp4 | HLS.js + DPlayer/ArtPlayer | 可作为预览播放器参考 |
| **倍速播放** | HTML5 Video API `playbackRate` | 原生API | 基础功能 |
| **广告过滤** | 解析时过滤广告片段 | 正则/关键词过滤 | 与创作无关 |
| **快捷键** | 空格暂停、方向键快进/快退 | 键盘事件监听 | 基础功能 |
| **响应式UI** | 适配PC端和移动端 | CSS Media Query + Flexbox/Grid | 前端通用方案 |
| **历史记录** | LocalStorage保存观看历史 | 浏览器本地存储 | 简单实现 |
| **收藏功能** | LocalStorage保存收藏列表 | 浏览器本地存储 | 简单实现 |

### 3.3 管理层应真正对标的MAM产品

**结论**: 若管理层的目标是影视资产管理，应对标以下开源方案，而非LibreTV。

#### 3.3.1 Nebula（开源广播自动化+MAM）

| 维度 | Nebula实现 | StarCanvas可借鉴点 |
|------|-----------|-------------------|
| **定位** | 电视台级广播自动化 + 媒体资产管理 | 专业级MAM参考 |
| **技术栈** | Python + React/TypeScript + Docker | 现代化全栈架构 |
| **元数据标准** | EBU Core标准 | 需建立元数据规范 |
| **编目体系** | 类型、编辑格式、氛围、版权、关联关系 | 需设计编目字段 |
| **转码能力** | FFmpeg + NVIDIA NVENC GPU加速 | 可直接复用FFmpeg管线 |
| **存储管理** | 代理文件（低分辨率预览）+ 母版文件分离 | 需设计存储策略 |
| **监控体系** | Prometheus + Grafana | 可直接复用 |
| **部署方式** | Docker容器化 | 推荐采用 |

**开源地址**: https://github.com/nebulabroadcast/nebula  
**许可**: GPL-3.0

#### 3.3.2 OpenWan（开源媒体资产管理系统）

| 维度 | OpenWan实现 | StarCanvas可借鉴点 |
|------|-----------|-------------------|

└── 服务层
    ├── 预览服务（流媒体播放 HLS/DASH）
    ├── 下载服务（分片下载/断点续传）

## 五、战略建议

### 5.1 短期（1-2个月）

1. **验证核心交互**: 优先实现无限画布 + 基础节点编辑器（参考React Flow）
2. **参数化面板**: 实现光影/镜头/景别三大核心面板，建立参数→Prompt映射引擎
3. **角色一致性**: 集成InstantID，解决角色外观一致性问题
4. **画风库建设**: 批量收集/训练20-50个核心画风LoRA

### 5.2 中期（3-6个月）

1. **AI Chat操作画布**: 集成LLM实现自然语言节点生成
2. **资产管理系统**: 实现@引用 + 个人/团队资产库
3. **一键拉片**: 集成PySceneDetect + CLIP实现视频自动拆解
4. **社区/克隆**: 基础工作流分享和克隆功能
5. **MAM基础版**: 若需资产管理，参考Nebula/Jellyfin搭建基础MAM

### 5.3 长期（6-12个月）

1. **专业控制深化**: 多视角锁定、720度全景、剪映导出
2. **Agent化**: 从工具升级为Agent，自主规划创作流程
3. **生态建设**: 创作者社区、KOL入驻、工作流市场
4. **企业级MAM**: 若服务B端，建设完整的Nebula级MAM系统
