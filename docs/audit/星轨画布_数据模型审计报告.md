## 四、核心业务数据链分析

### 当前数据流（存在瓶颈）

```
用户输入 → 前端 Zustand Store（内存） → CanvasDocument.nodes(Json)
                                                    ↓
                                         localStorage（4MB 限制）
```

**问题**：
- `CanvasDocument.nodes` 是万能 JSON 大杂烩，所有分镜/角色/场景/资产全混在一起
- 无法按 Shot 查询、无法按 Character 关联、无法按 Scene 分组
- 4MB localStorage 上限意味着大项目直接爆掉

### 目标数据流（需新增 6 张表）

```
Script ──→ StoryAnalysis ──→ StoryboardPlan ──→ CinematicShot ──→ ShotAsset
                │                    │                  │
                ↓                    ↓                  ↓
         CharacterBible        ContinuityReport    GenerationJob
         SceneBible
         VisualStyleBible
```

## 七、关键警示

1. **万能 JSON 反模式**：`CanvasDocument.nodes` 把所有业务数据（剧本/角色/分镜/资产）塞进一个 Json 字段，无法做关系查询、无法建索引、无法做数据完整性校验。这是当前架构最大的技术债务。
2. **前端类型 ≠ 数据模型**：`types/cinematic.ts` 的 25 字段 `CinematicShot` 定义得很好，但如果只存在于 TypeScript 编译时，运行时全部丢失。
3. **迁移风险**：一旦开始使用独立表，`CanvasDocument.nodes` JSON 数据需要做一次性迁移脚本，将嵌套在 JSON 中的 Shot/Character/Scene 数据提取到新表。
