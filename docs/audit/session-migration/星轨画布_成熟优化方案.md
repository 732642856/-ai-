### 🔥 维度一：性能与渲染优化（5项工具）

#### P0 · React Scan — 画布渲染性能诊断（0.5 天）

**为什么是 P0**：没有诊断数据就没有优化方向。React Scan 零代码侵入，跑一次就能看到画布所有组件的重渲染热力图。

```
$ npx -y react-scan@latest init
```

**具体操作**：
1. 在项目根目录执行初始化命令
2. 运行 `npm run dev`，打开画布页面
3. 拖拽节点、切换场景、执行 workflow，观察 React Scan 面板
4. 记录所有高亮标注的「不应重渲染但重渲染了」的组件
5. 按严重程度排序，作为后续 WDYR + React Flow 优化的目标

**验收标准**：找到至少 3 个性能瓶颈点，截图留档作为基准线。

#### P1 · why-did-you-render + React Flow 官方性能指南（2 天）

**为什么是 P1**：需基于 P0 的诊断数据，有针对地做深度优化。

**实施清单**：

| 步骤 | 操作 | 代码例 |
|------|------|--------|
| 1 | 给所有自定义节点组件加 `React.memo` | `const ContentNode = memo(({ data }) => ...)` |
| 2 | 给 `onNodeClick`、`onEdgeClick` 等回调加 `useCallback` | `const onNodeClick = useCallback(...)` |
| 3 | 避免直接订阅 `useStore((s) => s.nodes)`，改为独立字段 | 将 selectedNodeIds 存到独立 zustand store |
| 4 | 实施节点折叠机制 | 用 `node.hidden` 属性控制子节点显示 |
| 5 | 简化节点 CSS（阴影/动画/渐变） | 去除不必要的 CSS 特效 |
| 6 | 安装 `why-did-you-render` 验证修复效果 | `npm install @welldone-software/why-did-you-render` |

**验收标准**：
- React Flow 拖拽流畅度：节点数 ≤ 50 时 60fps，≤ 200 时 ≥ 30fps
- why-did-you-render 日志中无意外重复渲染报告

#### P2 · @next/bundle-analyzer — 包体积分析（0.5 天）

**为什么是 P2**：性能优化深入阶段，确保构建产物不膨胀。

```jsonc
// next.config.js
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
});
```

**预期收益**：识别重复依赖和未使用的 tree-shake 失败模块。

#### P2 · Biome — Rust 极速 Linter/Formatter（1 天）

**为什么是 P2**：替代 ESLint+Prettier 是工程效率优化，非功能性问题。

```bash
# 迁移
npx @biomejs/biome migrate eslint --write

# 使用
npx @biomejs/biome check --write .
```

**预期收益**：lint + format 时间缩短 50-80%。

### 🔍 维度二：可观测性与质量保障（4 项工具）

#### P0 · Sentry — 错误追踪 + AI 链路可观测性（1 天）

**为什么是 P0**：AI 创作链路长（剧本→拆 Shot→Bible→生图→视频），任何环节出错用户都无法得到预期结果，且错误难以复现。

**接入方案**：

```bash
npm install @sentry/nextjs
```

```typescript
// sentry.client.config.ts
import * as Sentry from '@sentry/nextjs';

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  tracesSampleRate: 1.0, // 开发期全量采样
  integrations: [Sentry.browserTracingIntegration()],
});

// 在关键 AI 步骤添加 span
// videoGenerationService.ts
Sentry.startSpan(
  { op: 'video.generate', name: 'Vidu SSE call' },
  async (span) => {
    // ... 现有的 SSE 调用逻辑
    span.end();
  }
);
```

**Span 埋点位置**：

| 步 | 文件 | 埋点名称 | 数据 |
|----|------|---------|------|
| 1 | `useScriptImport.ts` | `script.import` | 剧本长度、格式 |
| 2 | `useWorkflowRunner.ts` | `shot.breaking` | 拆出的 Shot 数 |
| 3 | `storyboard-service.ts` | `storyboard.image.gen` | 生成图片数、耗时 |
| 4 | `videoGenerationService.ts` | `video.generate` | backend、mode、耗时 |
| 5 | 导出模块 | `export.video` | 格式、时长 |

**验收标准**：能通过 Sentry 追踪任意用户的一次完整创作链路，定位到失败的具体步骤。

### 🧠 维度四：记忆与 AI 编排架构（4 项工具）

#### P0 · supermemory — 画布长期记忆引擎（2 天）

**为什么是 P0 核心**：这是当前「localStorage 竞态问题」的根治方案。supermemory 提供向量搜索 + SQLite 持久化，完美替代脆弱的 `localStorage`。

**架构设计**：

```
┌─────────────────────────────────────────────────┐
│                  supermemory                       │
│  ┌────────────┐  ┌──────────┐  ┌──────────────┐  │
│  │  Vector DB  │  │  SQLite  │  │   Semantic   │  │
│  │  (similar)  │  │(exact)   │  │    Search    │  │
│  └────────────┘  └──────────┘  └──────────────┘  │
└──────────────────────┬──────────────────────────┘
                       │ MCP Server / REST API
┌──────────────────────▼──────────────────────────┐
│                  StarCanvas                        │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐ │
│  │ Canvas   │ │  Chat    │ │  Project Bible   │ │
│  │ State    │ │  History │ │  (characters,    │ │
│  │(nodes/   │ │          │ │   scenes, props) │ │
│  │ edges)   │ │          │ │                  │ │
│  └──────────┘ └──────────┘ └──────────────────┘ │
└─────────────────────────────────────────────────┘
```

**接入方案（MCP Server 模式，最小侵入）**：

```bash
# 1. 启动 supermemory MCP Server
npx supermemory mcp-server
```

```typescript
// 2. 封装超级记忆客户端
// lib/supermemory.ts
class SuperMemoryClient {
  private baseUrl: string;
  
  constructor() {
    this.baseUrl = process.env.SUPERMEMORY_URL || 'http://localhost:3001';
  }
  
  // 存储画布状态
  async saveCanvasState(projectId: string, state: CanvasState) {
    return fetch(`${this.baseUrl}/memory`, {
      method: 'POST',
      body: JSON.stringify({
        key: `canvas:${projectId}`,
        content: JSON.stringify(state),
        metadata: { type: 'canvas_state', projectId },
      }),
    });
  }
  
  // 语义检索项目记忆
  async searchMemory(projectId: string, query: string) {
    const res = await fetch(`${this.baseUrl}/search`, {
      method: 'POST',
      body: JSON.stringify({
        query,
        filter: { projectId },
      }),
    });
    return res.json();
  }
}

// 3. 在 useCanvasPersistence.ts 中使用
// 替换当前 localStorage 逻辑
const supermemory = new SuperMemoryClient();

useEffect(() => {
  if (!initialRestoreDone && nodes.length > 0) {
    // 优先从 supermemory 恢复
    const saved = await supermemory.searchMemory(projectId, 'latest canvas state');
    if (saved.length > 0) {
      setNodes(JSON.parse(saved[0].content));
    }
    setInitialRestoreDone(true);
  }
}, [projectId]);
```

**存储内容映射**：

| 数据类型 | 当前存储 | 迁移后存储 | supermemory 索引 |
|---------|---------|-----------|-----------------|
| Canvas 节点/边 | localStorage | supermemory | key: `canvas:<projectId>` |
| Chat 历史 | localStorage / memory | supermemory | key: `chat:<projectId>:<sessionId>` |
| Project Bible | 内存 + API | supermemory | key: `bible:<projectId>` |
| 用户偏好 | localStorage | supermemory | key: `prefs:<userId>` |
| AI 生成历史 | 浏览器内存 | supermemory | 语义搜索，自然语言检索 |

**多会话语义检索示例**：

```typescript
// 用户说："之前讨论的那个雨夜场景的参数是什么？"
const results = await supermemory.searchMemory(projectId, '雨夜场景 拍摄参数 曝光设置');
// 直接返回匹配的历史记录
```

**验收标准**：
- localStorage 竞态问题彻底消除
- 刷新页面后画布状态完全恢复
- 支持自然语言检索历史创作记忆
- 页面关闭/崩溃后数据不丢失

#### P3 · ECC — Agent 工程哲学参考（0.5 天）

ECC (Efficient Compound Capability) 的核心理念：

1. **Skill 系统**：可组合的能力单元
2. **本能（Instinct）**：Agent 的基础行为模式
3. **记忆（Memory）**：跨会话上下文
4. **安全（Safety）**：边界约束

**对星轨画布的影响**：在设计"制片层""管理层"时，可以参考 ECC 的 Agent 分层架构，而不是把业务逻辑全部写在 React hooks 里。

#### P3 · compound-engineering-plugin — 开发流程参考（0.5 天）

核心理念：**规划→执行→评审→沉淀**。

```
Plan   →   Execute   →   Review   →   Settle
┌────┐   ┌────────┐   ┌────────┐   ┌──────┐
│PRD │→  │ 代码实现 │→  │ Code   │→  │文档/  │
│设计 │   │        │   │ Review │   │最佳   │
│     │   │        │   │        │   │实践   │
└────┘   └────────┘   └────────┘   └──────┘
```

**建议**：把这个循环模式引入星轨画布的 GitHub Actions 工作流，每次 PR 自动跑完整质量门禁。

| 画布渲染 | FPS（200 节点） | 约 15-20 | ≥30fps | 节点折叠 + memoization |

## 五、风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| supermemory 项目迭代快，API 不稳定 | 集成后频繁适配 | 锁定版本，写适配层隔离 |
| Sentry 免费额度有限 | 超额付费 | 先用免费版，超标后切换 Highlight.io |
| Biome 迁移可能破坏现有格式化 | 大量 diff | 先在 CI 中并行运行，对比无差异后再切换 |
| MarkItDown 147k 项目过于通用 | 对中文剧本排版可能不完美 | 加后处理 step 优化中文格式 |
| taste-skill 设计理念过于激进 | 偏离用户习惯 | 渐进式改进，不做一次性大改版 |

## 六、总结

本方案将 **16 个开源工具** 融入星轨画布项目的四个维度，分为三阶段实施：

**第一阶段（P0，第 1 周）**：用 React Scan 诊断性能瓶颈，用 Sentry 建立 AI 链路可观测性，引入 taste-skill 设计理念，用 supermemory 彻底解决 localStorage 问题——这四个是**投入产出比最高的，做完就能感到质的改变**。

**第二阶段（P1-P2，第 2-4 周）**：深度优化 React Flow 渲染性能，加固 E2E 测试稳定性，扩展文件导入格式，引入 AI 代码审查，参考视频导出管线。

**第三阶段（P3，第 5-8 周）**：设立性能/质量预算门禁，引入多 Agent 架构框架，深化安全扫描。

**最建议优先动手的三件事**：
1. **supermemory → 替换 localStorage**（根治竞态，为 Project Bible 和长期记忆铺路）
2. **React Scan → 诊断画布性能**（为后续深度优化提供数据）
3. **Sentry → 接入 AI 链路追踪**（创作流程长了，出错定位不能靠猜）
